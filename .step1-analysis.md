## Bug 1 ‚Äî Online Multiplayer: Host stuck on waiting, guests can't join

### User Impact
Host enters name, clicks "Room Banao", reaches lobby with only self shown. Guest enters same room code, sees "Host ke shuru karne ka intezaar..." indefinitely. No error shown on either side. Game never starts. User has no way to diagnose ‚Äî the `CircularProgress` spinner in `LobbyScreen.jsx:68-73` provides zero feedback on connection state.

### Root Cause Analysis
The online multiplayer uses **WebSocket ‚Üí Cloudflare Worker** (`wss://dhandha-multiplayer.workers.dev`). Flow:

1. `MultiplayerSetupScreen` ‚Üí `handleRoomReady` ‚Üí `mp.connect(roomCode)` ‚Üí `new WebSocket(\`${base}/${roomCode}\`)`
2. Guest does same ‚Üí sends `HELLO` after 600ms (`App.jsx:179`)
3. Host's `handleMessage` receives `HELLO` ‚Üí adds player ‚Üí broadcasts `ROSTER`

**Three concrete failure modes (any one causes the bug):**

#### a) WebSocket room routing is broken (most likely)
The Cloudflare Worker at `wss://dhandha-multiplayer.workers.dev` must implement room-based pub/sub (route messages by room code). If the worker is not deployed, returns 404, or doesn't properly segregate rooms, host and guest never see each other's messages. The 600ms `setTimeout` for `HELLO` (`App.jsx:179`) is a race condition ‚Äî if the WebSocket isn't `OPEN` yet, the message is silently lost.

#### b) No visible connection error in lobby
`useMultiplayer.js:29` sets `error` on WebSocket failure, but **`LobbyScreen` never reads `error` or `connected` states**. A failed WebSocket shows the same spinner as a healthy waiting state. User cannot distinguish "waiting for player" from "network broken".

#### c) No auto-reconnect
If the WebSocket drops after initial connect (common on mobile networks), `onclose` fires, `connected` becomes `false`, but nothing reconnects and no UI feedback changes. Lobby stays frozen forever.

### Edge Cases
- Worker URL env var `VITE_WS_URL` may differ between dev/prod; if set incorrectly in production build, it silently overrides the default.
- Room code collision: `generateCode()` produces 4-char uppercase alphanumeric (36‚Å¥ ‚âà 1.6M combos). Two concurrent rooms with same code merge invisibly.
- Guest's 600ms `setTimeout` is hardcoded ‚Äî a slow 3G connection will miss it.

### Fix Recommendation
1. **Show WebSocket status in LobbyScreen** ‚Äî display `mp.connected` and `mp.error` so user knows if connection failed
2. **Send HELLO only after confirming `ws.readyState === OPEN`** ‚Äî move from timeout-based to event-driven
3. **Add reconnection logic** with exponential backoff in `useMultiplayer`
4. **Add a "Refresh" / "Try Again" button** in lobby for both host and guest
5. If worker is unreliable, add a fallback path (direct IP relay)

## Bug 2 ‚Äî Wild Card Colour Lock: Can never change colour after play

### User Impact
Player plays a dual-colour wild (e.g., Pink/Orange) and picks Pink at play time. Next turn, they realize Orange would help complete a set. Real Monopoly Deal rules say **you CAN change a wild card's colour on your turn**. Current code permanently locks it to the original choice. This is a straight rules violation.

### Root Cause
`src/game/gameLogic.js:225-228`:
```js
if (card.type === CARD_TYPES.WILD_PROPERTY && targetColor) {
  color = targetColor
  card.color = targetColor   // ‚Üê MUTATES the card object permanently!
}
```
The `card.color` property is overwritten from its original `colors` array (e.g., `['pink', 'orange']`) to the single chosen colour (`'pink'`). After this, the original dual-colour info is lost forever.

### Edge Cases
- Full wild (`colors: ['wild']`) assigned to Brown ‚Äî can never switch to Light Blue later, even if Brown set is full.
- After colour lock, wild card effectively behaves like a regular property with no indication it was ever a wild.
- When set is broken and rebuilt (via Forced Deal, etc.), the locked colour persists incorrectly.

### Fix Recommendation
**Don't mutate `card.color`.** Instead:
- Keep `card.color` = original default (e.g., `colors[0]`)
- Add a `currentColor` or `assignedColor` field on the card that the game checks when determining set membership
- On current player's turn, allow re-selecting `assignedColor` via a UI control (tap the wild card on board ‚Üí colour picker)
- `playPropertyCard` should use a separate field to track assignment without overwriting the original

## Bug 3 ‚Äî Deal Breaker: Instantly steals set, never prompts for Just Say No

### User Impact
Victim has a complete set and holds a JSN card in hand. Attacker plays Deal Breaker. The set is stolen instantly with **zero chance to play JSN**. The victim watches helplessly. This breaks a core mechanic of the game ‚Äî JSN exists specifically to counter Deal Breaker.

### Root Cause
`src/game/useGameState.js:419-457` (`DEAL_BREAKER_STEAL` case):

The reducer checks for `victim.insurance` (custom card) and auto-cancels if present (line 428-435), but **there is no code path that checks for JSN or transitions to `PHASE.ACTION_RESPONSE`** to give the victim an opportunity.

Compare with other actions:
- **Debt Collector** ‚Üí `DEBT_COLLECTOR` case ‚Üí sets `phase: PHASE.ACTION_RESPONSE` ‚Üí `PaymentSheet` renders JSN button
- **Birthday** ‚Üí `BIRTHDAY` case ‚Üí same pattern ‚Üí JSN prompt works
- **Deal Breaker** ‚Üí `DEAL_BREAKER_SELECT` (player picks target) ‚Üí dispatches `DEAL_BREAKER_STEAL` ‚Üí **directly executes theft**

The `DEAL_BREAKER_STEAL` action is dispatched from `ActionModal.jsx:262` with **no intermediate phase**. The victim never sees a "Nahi!" button.

### Edge Cases
- Victim has JSN but also has Insurance ‚Äî Insurance handles it anyway, but JSN should also work
- Multiple victims (only one target per Deal Breaker, so N/A)
- Original actor could counter-JSN the victim's JSN ‚Äî this chain is entirely missing

### Fix Recommendation
Restructure `DEAL_BREAKER_STEAL` to route through `PHASE.ACTION_RESPONSE`:

1. After `DEAL_BREAKER_SELECT` (player picks target set), **don't execute steal yet**
2. Set `phase: PHASE.ACTION_RESPONSE` with `pendingAction` holding the Deal Breaker details
3. `PaymentSheet`-like UI for victim showing "Nahi!" button (JSN)
4. If victim doesn't JSN ‚Üí execute the steal
5. If victim JSNs ‚Üí normal JSN chain (original actor can counter-JSN)

Current Insurance check (`victim.insurance`) should remain as a **pre-JSN auto-block** ‚Äî Insurance is consumed, no JSN prompt needed.

## Bug 4 ‚Äî Rent Payment Modal Hides Player's Hand

### User Impact
Player owes ‚Çπ5Cr rent. They have ‚Çπ3Cr in bank, a property worth ‚Çπ4Cr on board, AND a ‚Çπ5Cr money card in hand. The `PaymentSheet` only shows bank + property + buildings. The hand card is invisible. Player may unnecessarily give away a property when they could have paid from hand.

In real Monopoly Deal rules, **any card from hand can be used as payment at its face value**.

### Root Cause
`src/components/game/ActionModal.jsx:444-617` ‚Äî `PaymentSheet` builds `allAssets` from:
```js
const cashAssets = [...payer.bank].map(...)        // line 448
const propAssets = orderPropertyColors(payer.properties).flatMap(...)  // line 449-450
const buildingAssets = [...]  // lines 452-463
```
**`payer.hand` is never included.** The `allAssets` array (line 464) only contains banked money, board properties, and buildings.

The `setTotal` / payment calculation relies entirely on `allAssets`. Hand cards are invisible and unusable for payment.

### Edge Cases
- Player has zero bank and zero properties but multiple high-value money cards in hand ‚Üí forced to "Pass" even though they CAN pay
- Player has one property on board worth ‚Çπ2Cr and ‚Çπ4Cr money card in hand. They need ‚Çπ5Cr. They give away the property (‚Çπ2Cr) and can't find the other ‚Çπ3Cr ‚Üí game soft-locks them (can't meet payment)
- Discard phase just ended ‚Üí hand is full (7 cards) ‚Üí none visible during payment

### Fix Recommendation
Add hand cards to the `PaymentSheet` assets:

```js
const handAssets = payer.hand
  .filter(c => c.type !== CARD_TYPES.PROPERTY && c.type !== CARD_TYPES.WILD_PROPERTY)
  .map(c => ({ ...c, _from: 'hand' }))
```

And merge into `allAssets`. Property cards in hand should be excluded (per real rules, properties on board can be given, but properties in hand haven't been played yet ‚Äî need to check whether real rules allow giving unplayed properties).

Also: `<CounterpartyStrip>` (line 542) shows the creditor's holdings, which is helpful. The hand should be displayed above the assets list with proper labelling so player can see ALL payment options at once.

## Bug 5 ‚Äî Rent Card Playable with Zero Matching Properties

### User Impact
Player selects a Rent card (e.g., Red/Yellow) but has no Red or Yellow properties. The "Rent Maango" button appears in `PlayOptions`. Clicking it opens `RentSelector` with **zero colour options displayed**. Player is confused ‚Äî why did the app let them select this card? The button should not appear.

For Wild Rent cards, the player CAN select any colour they own. But if they own zero properties at all, same problem.

In multiplayer, a hacked client could dispatch `PLAY_RENT` with a fake `targetColor` and the server would execute it (rent = ‚Çπ0, wastes a play).

### Root Cause
**UI layer** (`GameScreen.jsx:368-371`):
```jsx
{isRent && (
  <Button ... onClick={onPlayRent}>
    üí∞ Rent Maango
  </Button>
)}
```
The button shows unconditionally for any rent card. No guard checks if the player has eligible properties.

**Reducer layer** (`useGameState.js:200-237`):
```js
case 'PLAY_RENT': {
  const playerProps = player.properties[targetColor] || []
  const count = playerProps.length
  const baseAmount = getRentForColor(targetColor, count, player.buildings)
  ...
}
```
No validation that `playerProps.length > 0`. `getRentForColor` with `count=0` ‚Üí `set.rentValues[-1]` ‚Üí `undefined || 0` ‚Üí rent = ‚Çπ0. Action executes uselessly and burns a play.

**UI selector layer** (`GameScreen.jsx:401-403`):
```js
const eligibleColors = Object.keys(currentPlayer.properties).filter(color => {
  if (!card.wild) return card.colors.includes(color)
  return true
})
```
For non-wild rent, if player has zero matching colours, `eligibleColors = []`. Chips don't render, but the user already navigated into a dead-end flow.

### Edge Cases
- Player plays a Rent card with matching properties but they're all in incomplete sets ‚Üí rent = base value (no bonus), which is valid
- Player uses "Double The Rent" first, then plays Rent with zero matching properties ‚Üí doubleRentActive flag is consumed (`s.doubleRentActive = false` at line 217) but rent = 0 √ó 2 = 0. Double Rent is wasted.
- Two-player game: non-wild rent targets all other players (just one). Wild rent targets one. With zero properties, both paths produce ‚Çπ0 rent.

### Fix Recommendation
1. **`PlayOptions`**: Only show "Rent Maango" button when player has ‚â•1 eligible property for this rent card
2. **`RentSelector`**: Guard against empty `eligibleColors` ‚Äî show an error Chip ("Pehle is colour ki property lagao") instead of a blank panel
3. **`PLAY_RENT` reducer**: Add validation:
   ```js
   const playerProps = player.properties[targetColor] || []
   if (playerProps.length === 0) return state  // can't rent without properties
   ```
4. **`doubleRentActive` flag**: Don't consume it (`s.doubleRentActive = false`) until rent amount is confirmed > 0
